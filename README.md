# ЁЯМжя╕П Weather Bookmark API

A production-ready REST API built with **FastAPI**, **SQLModel**, and **PostgreSQL** (Supabase) for managing city weather bookmarks тАФ now with an **AI-powered weather assistant** backed by Groq + pydantic-ai.

Features in-memory caching, per-IP rate limiting, concurrent weather fetching, weather history with cursor pagination, temperature alerts, persistent AI chat sessions with full conversation history, and schema migration via Alembic.

## тЪЩя╕П Setup

### 1. Clone & create virtual environment

```bash
git clone <repo-url>
cd weather-services-api
python -m venv .venv
source .venv/scripts/activate   # Windows (bash)
# source .venv/bin/activate     # macOS/Linux
```

### 2. Install dependencies

```bash
uv pip install fastapi sqlmodel alembic httpx fakeredis pydantic-settings psycopg2-binary pydantic-ai groq
```

### 3. Configure environment variables

Create a `.env` file in the project root:

```env
OPENWEATHER_API_KEY=your_openweather_key
DATABASE_URL=postgresql://user:password@host:port/dbname
CACHE_TTL_SECONDS=600
RATE_LIMIT_MAX_REQUESTS=10
RATE_LIMIT_WINDOW_SECONDS=60
TIMEOUT=10.0
GROQ_API_KEY=your_groq_api_key
LLM_MODEL=groq:llama-3.3-70b-versatile
```

> **Supabase tip:** Use the **Session Pooler** URL (`aws-0-[region].pooler.supabase.com` on port `6543`) rather than the direct connection URL to avoid IPv6 DNS resolution issues on local networks.
>
> **Supabase free tier:** Projects auto-pause after ~1 week of inactivity. If you get a DNS error on both URLs simultaneously, go to your [Supabase dashboard](https://supabase.com/dashboard) and click **Restore**.

### 4. Run database migrations

Schema is fully managed by Alembic тАФ never created at runtime.

```bash
# First-time setup (if tables already exist from create_all):
alembic revision -m "Initial baseline"
alembic stamp head

# Apply all pending migrations:
alembic upgrade head
```

### 5. Start the server

```bash
fastapi dev main.py
```

API docs available at: `http://localhost:8000/docs`

## ЁЯЧДя╕П Database Migrations (Alembic)

Schema changes are a deliberate CLI operation тАФ the app never modifies the schema at runtime.

### Daily workflow

```bash
# 1. Edit your model in models.py or ai_layer/ai_models.py
# 2. Generate a migration
alembic revision --autogenerate -m "describe your change"

# 3. Review the generated file in alembic/versions/ тАФ always check before applying!
# 4. Apply it
alembic upgrade head
```

### Useful commands

```bash
alembic current          # Show current DB revision
alembic history          # Show full migration history
alembic downgrade -1     # Roll back one migration
alembic downgrade base   # Roll back all migrations
```

> тЪая╕П **Always review autogenerated migrations before applying.** When adding a non-nullable column to a table with existing rows, manually add `server_default`:
> ```python
> op.add_column('bookmark', sa.Column('is_favorite', sa.Boolean(), server_default=sa.false(), nullable=False))
> ```

### ЁЯдЦ AI Chat

Conversational AI weather assistant powered by **Groq** and **pydantic-ai**. The agent has access to your saved bookmarks, live weather data, temperature alerts, and historical trends.

| Method | Endpoint | Description |
|---|---|---|
| `POST` | `/v1/ai/chat` | Send a message to the AI weather assistant |
| `DELETE` | `/v1/ai/chat/{session_id}` | Clear conversation history for a session |

#### `POST /v1/ai/chat`

**Request body:**

```json
{
  "prompt": "Which of my bookmarked cities is the hottest right now?",
  "sessionId": "optional-uuid-to-continue-a-conversation"
}
```

- **Omit `sessionId`** to start a new conversation тАФ a new UUID is returned in the response.
- **Include `sessionId`** from a previous response to continue the conversation with full history.

**Response:**

```json
{
  "reply": "Lagos is currently the hottest at 34┬░C...",
  "sessionId": "b87b2524-096c-4f22-903e-c9ff4476aff4"
}
```

**What the AI can do:**

- List your saved bookmarks
- Fetch live or cached weather for any city
- Check which bookmarks are currently exceeding their temperature threshold
- Compare weather across multiple cities concurrently
- Analyse historical temperature trends and anomalies
- Answer follow-up questions using full conversation context

**Error responses:**

| Code | Meaning |
|---|---|
| `200` | Successful AI response |
| `422` | Invalid request (e.g. empty prompt) |
| `503` | AI service temporarily unavailable тАФ retry with the same `sessionId` |

## ЁЯПЫя╕П Architecture

### Service Layer

| Service | Responsibility |
|---|---|
| `WeatherCacheService` | fakeredis-backed Cache-Aside pattern + per-IP rate limiting |
| `WeatherApiService` | httpx async calls to OpenWeatherMap, orchestrates cache reads/writes |
| `WeatherHistoryService` | Persists weather snapshots to DB, provides cursor-paginated reads |
| `AIOrchestrator` | Manages agent execution, Redis response caching, DB history persistence, and observability logging |

### AI Layer

```
POST /v1/ai/chat
       тФВ
       тЦ╝
  AIOrchestrator.handle_chat()
       тФВ
       тФЬтФА 1. Check Redis cache (prompt hash тЖТ cached reply)
       тФВ
       тФЬтФА 2. Load conversation history from PostgreSQL (ChatMessage)
       тФВ
       тФЬтФА 3. Run pydantic-ai Agent (Groq LLM)
       тФВ         тФВ
       тФВ         тФЬтФА Tool: get_my_bookmarks       тЖТ DB query
       тФВ         тФЬтФА Tool: get_weather_for_city   тЖТ Cache-Aside тЖТ OpenWeather API
       тФВ         тФЬтФА Tool: check_temperature_alerts тЖТ concurrent weather fetches
       тФВ         тФФтФА Tool: get_weather_trends     тЖТ historical DB analysis
       тФВ
       тФЬтФА 4. Save new messages to PostgreSQL (ChatMessage)
       тФВ
       тФЬтФА 5. Log execution to PostgreSQL (AILog) тАФ latency, tokens, tools used
       тФВ
       тФФтФА 6. Cache response in Redis тЖТ return reply
```

### Caching (Cache-Aside)

```
Request тЖТ Check fakeredis тЖТ HIT: return cached data (cached: true)
                          тЖТ MISS: call OpenWeather API тЖТ store in cache тЖТ return (cached: false)
```

Cache TTL is controlled by `CACHE_TTL_SECONDS` in `.env` (default: 600s / 10 min).

### Rate Limiting

Per-IP rate limiting using atomic Redis `INCR` on a rolling time-window key. Returns `HTTP 429` with a `Retry-After` header when exceeded. Configurable via `RATE_LIMIT_MAX_REQUESTS` and `RATE_LIMIT_WINDOW_SECONDS`.

### ETag Caching

`GET /v1/bookmarks/{id}` computes a SHA-256 hash of the response body and returns it as an `ETag` header. If the client sends `If-None-Match` with the same hash, it receives `304 Not Modified` with no body.
